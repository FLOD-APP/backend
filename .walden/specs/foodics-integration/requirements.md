# Requirements Document

## Introduction

FLOD uses Foodics as its POS system across 15 branches in Riyadh. Currently, **all orders are entered manually by cashiers daily** -- no API connection exists. Ferotech (previous dev company) never built the integration, making this greenfield work. The integration is a P0 blocker (BL-009 backend, BL-095 frontend) because:

1. **ZATCA Phase 2 e-invoicing** runs through Foodics -- the app must push orders there to maintain tax compliance
2. **Wallet/balance model** requires Foodics House Account sync for financial reconciliation
3. **Abu Talin (Finance)** needs same-day sync for inventory, sales closing, and reporting

**Design principle:** Foodics sync is **eventual, not transactional**. FLOD's database is the source of truth. Foodics is kept in sync via best-effort + retry. Foodics downtime does NOT block meal collections.

**Blocked items:** API keys must be generated by Abu Talin from his Foodics admin dashboard. Sandbox access is pending (BL-110). Phase 0 (this spec) covers all work achievable without API keys.

**API reference:** Foodics POS API v5 at `https://api.foodics.com/v5` (production) / `https://api-sandbox.foodics.com/v5` (sandbox). Public docs at docs.foodics.com.

## Requirements

### R1 Foodics HTTP Client

**User Story:** As a backend developer, I want a typed HTTP client for the Foodics API v5, so that all API communication is centralised with rate limiting, retry, and error handling.

#### Acceptance Criteria

1. `R1.AC1` WHEN the client sends a request to the Foodics API, the system SHALL include `Authorization: Bearer {token}` and `Content-Type: application/json` headers
2. `R1.AC2` WHEN the Foodics API returns a 429 rate limit response, the system SHALL wait for the duration specified in the `retry-after` header before retrying
3. `R1.AC3` WHEN the Foodics API returns a 5xx server error, the system SHALL retry with exponential backoff and jitter (max 3 retries)
4. `R1.AC4` WHEN the client has sent 30 requests within the current minute, the system SHALL queue subsequent requests until the rate limit window resets (token bucket algorithm)
5. `R1.AC5` WHEN a Zod response schema is provided, the system SHALL validate the API response against it and throw a `FoodicsValidationError` on mismatch
6. `R1.AC6` WHEN the Foodics API returns a 4xx/5xx error, the system SHALL throw a `FoodicsApiError` with the original HTTP status, error message, and a `retryable` flag
7. `R1.AC7` The system SHALL log all Foodics API requests and responses via Pino structured logging

### R2 Foodics Order Builder

**User Story:** As the collection service, I want to transform FLOD meal collection data into a Foodics order payload, so that orders are correctly structured for the Foodics API.

#### Acceptance Criteria

1. `R2.AC1` WHEN a meal is collected, the system SHALL build a Foodics order with `type: 3` (pickup), the correct `branch_id`, `customer_id`, and `reference: FLOD-{mealId}`
2. `R2.AC2` WHEN a meal has a VAT-inclusive price (e.g. SAR 24), the system SHALL extract VAT correctly (`24 / 1.15 = 20.87 ex-VAT, 3.13 VAT`) and set `unit_price` to the VAT-inclusive amount as a 5-decimal string
3. `R2.AC3` WHEN a meal has modifier options (carb, sauce, vegetables), the system SHALL include them in the `options` array with `modifier_option_id` and `quantity: 1`
4. `R2.AC4` WHEN a meal has customer notes (e.g. "No onion"), the system SHALL include them in the product's `notes` field
5. `R2.AC5` WHEN a meal has a per-line discount, the system SHALL include `discount_amount` on the product and reduce the payment amount accordingly
6. `R2.AC6` WHEN `customer_id` is not available, the system SHALL omit it from the order (Foodics will skip customer association)
7. `R2.AC7` The order builder SHALL be a pure function with no I/O dependencies, fully unit-testable

### R3 Foodics Sync Service

**User Story:** As the system, I want to sync meal collections to Foodics with automatic retry on failure, so that all orders eventually appear in Foodics even during API downtime.

#### Acceptance Criteria

1. `R3.AC1` WHEN a meal is collected and Foodics sync is enabled, the system SHALL attempt to create a Foodics order via `POST /orders`
2. `R3.AC2` WHEN the Foodics order creation succeeds, the system SHALL store the `foodics_order_id` and `foodics_synced_at` on the `subscription_daily_meals` record and log success in `foodics_sync_log`
3. `R3.AC3` WHEN the Foodics order creation fails with a retryable error, the system SHALL create a retry entry in `foodics_sync_log` with exponential backoff (immediate, 1m, 5m, 15m, 1h)
4. `R3.AC4` WHEN the Foodics order creation fails with a non-retryable error (e.g. 400 validation), the system SHALL mark the sync as `failed` permanently
5. `R3.AC5` WHEN the retry queue processor runs, the system SHALL pick up entries where `next_retry_at <= now` and `status = 'retrying'` and attempt to resync them
6. `R3.AC6` WHEN a sync entry has exhausted all retry attempts (default 5), the system SHALL mark it as `failed` permanently, requiring manual intervention
7. `R3.AC7` The system SHALL provide a feature flag `FOODICS_SYNC_ENABLED` that gates the entire sync -- when `false`, no Foodics API calls are made
8. `R3.AC8` Foodics sync failure SHALL NOT block or roll back the FLOD meal collection transaction

### R4 Database Schema for Foodics Tracking

**User Story:** As the system, I want database columns and tables to track Foodics sync state, so that every meal's Foodics status is auditable.

#### Acceptance Criteria

1. `R4.AC1` The `branches` table SHALL have a `foodics_branch_id` text column mapping FLOD branches to Foodics branch UUIDs
2. `R4.AC2` The `products` table SHALL have a `foodics_product_id` text column mapping FLOD products to Foodics product UUIDs
3. `R4.AC3` The `users` table SHALL have a `foodics_customer_id` text column mapping FLOD users to Foodics customer UUIDs
4. `R4.AC4` The `subscription_daily_meals` table SHALL have `foodics_order_id` (text), `foodics_synced_at` (timestamptz), and `foodics_sync_status` (enum: pending/synced/failed/retrying) columns
5. `R4.AC5` The system SHALL have a `foodics_sync_log` table with: `id` (uuid PK), `meal_id` (FK to subscription_daily_meals), `status` (foodics_sync_status enum), `foodics_order_id`, `attempt` (int), `error_message` (text), `request_payload` (jsonb), `next_retry_at` (timestamptz), `created_at`, `updated_at`
6. `R4.AC6` The `foodics_sync_log` table SHALL have indexes on `(status, next_retry_at)` for retry queue queries and on `(meal_id)` for audit lookups

### R5 Admin Monitoring Endpoints

**User Story:** As Abu Talin (Finance), I want to monitor Foodics sync status and manually retry failed syncs, so that I can ensure all orders are reconciled.

#### Acceptance Criteria

1. `R5.AC1` The system SHALL expose `GET /admin/foodics/sync-status` returning counts by status (pending, synced, retrying, failed)
2. `R5.AC2` The system SHALL expose `POST /admin/foodics/retry/:id` allowing manual retry of a specific failed sync entry
3. `R5.AC3` Admin endpoints SHALL require authentication and admin role

### R6 Foodics API Type Safety

**User Story:** As a backend developer, I want TypeScript interfaces and Zod validators for all Foodics API shapes, so that I catch API contract violations at compile time and runtime.

#### Acceptance Criteria

1. `R6.AC1` The system SHALL have TypeScript interfaces for all Foodics API request/response shapes (branches, products, categories, customers, house accounts, payment methods, taxes, orders, orders calculator, webhooks, errors)
2. `R6.AC2` The system SHALL have Zod schemas for runtime validation of Foodics API responses that FLOD consumes
3. `R6.AC3` The system SHALL have generic `foodicsPaginatedSchema()` and `foodicsSingleSchema()` wrappers for paginated and single-entity responses

### R7 Configuration and Feature Flags

**User Story:** As a DevOps engineer, I want all Foodics configuration in environment variables with safe defaults, so that Foodics sync can be enabled/disabled without code changes.

#### Acceptance Criteria

1. `R7.AC1` The system SHALL read `FOODICS_API_URL` (base URL), `FOODICS_ACCESS_TOKEN`, `FOODICS_PAYMENT_METHOD_ID`, `FOODICS_SYNC_ENABLED` (boolean), and `FOODICS_MAX_RETRY_ATTEMPTS` (number) from environment variables
2. `R7.AC2` `FOODICS_SYNC_ENABLED` SHALL default to `false` so Foodics sync is opt-in
3. `R7.AC3` `FOODICS_MAX_RETRY_ATTEMPTS` SHALL default to `5`
4. `R7.AC4` `.env.example` SHALL document all Foodics environment variables with comments

## Non-Functional Requirements

- `NFR1` The Foodics HTTP client SHALL respect a rate limit of 30 requests per minute per token per IP
- `NFR2` The order builder SHALL execute in < 1ms (pure computation, no I/O)
- `NFR3` Retry queue processing SHALL handle batches of 10 entries per cycle to respect Foodics rate limits
- `NFR4` All Foodics module code SHALL pass TypeScript strict mode with no `any` types

## Constraints And Dependencies

- `C1` Foodics API v5 at `https://api-sandbox.foodics.com/v5` (sandbox) / `https://api.foodics.com/v5` (production)
- `C2` Node.js 20+ native `fetch` -- no axios dependency for the Foodics client
- `C3` Drizzle ORM 0.45 for all database operations
- `C4` Zod 4 for runtime validation
- `C5` Pino 10 for structured logging
- `C6` Foodics API keys pending from Abu Talin (BL-110) -- blocks sandbox/production testing
- `C7` Six unanswered business questions pending from Abu Talin (BL-113) -- does not block Phase 0 code
